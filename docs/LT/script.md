# LT台本：GitHub Actions+VitePressで読み返しやすいメモを取る（5分想定）

## [00:00 - 00:30] 1. 導入：メモの価値と課題
こんにちは。
今日は「GitHub ActionsとVitePressで、読み返しやすいメモ環境を作る」というテーマでお話しします。

自分は、メモは「書いたら終わり」ではなく、「読み返す」ことで初めて価値があると思っています。
ただ読み返さない雑記を重ねるなんて、「書き散らし」にすぎませんからね。

また、昨今ObsidianやCoing Agentのようなツールが台頭している中で、書きやすく、参照しやすいMarkdown形式のメモの重要性が増していると感じています。

Twitter/Xに流すほど軽くなく、
Qiitaの記事ほど重くもない、
そういう「個人的だけど整理されたメモ」こそが、いざ読み返すときに一番扱いやすい情報となると思っています。

これらの「中間」にある、自分にとって一番価値のある情報を、ローカルとWeb公開の両立で管理したい。
これが今回のモチベーションです。

## [00:30 - 01:30] 2. VitePress＋GitHub Actionsを選ぶ理由
なぜこの組み合わせなのか。理由はシンプルに「手間を減らし、かつ表現力を維持したい」からです。

- **VitePress**：MarkdownをHTMLに変換してくれるVueベースのツールです。最近のOSSドキュメントはだいたいこれかVuePressですね。
- **GitHub Actions**：これを組み合わせることで、「プッシュしたら勝手にデプロイされる」環境が手に入ります。

ローカルでプレビューしながら書き、`git push` するだけでWeb上のドキュメントも更新される。このシンプルさが継続のコツです。
また、我々学生はGitHub Educationを利用できるので、プライベートリポジトリであっても無料でGitHub Actionsを使えるのが大きなメリットですね。

## [01:30 - 03:00] 3. ワークフローと設定
次に、具体的な構成を見ていきましょう。

（スライドの構成図を見せながら）
リポジトリ構成はNodeプロジェクトの標準的な形で、導入も非常にシンプルです。
`docs/` フォルダをVitePressのルートに設定するだけで、既存プロジェクトにも簡単に導入できます。

そして大事なのがデプロイの自動化。GitHub Actionsのワークフロー定義もシンプルです。
実際に行われていることは、本質的には3ステップだけ：

1. コード取得とセットアップ
2. `vitepress build` を実行
3. 生成された成果物を GitHub Pages に自動デプロイ

これだけです。スライドのYAMLは長く見えますが、やっていることはシンプルなんです。
プッシュボタンを押したら、Web上に自動で反映される——この体験が継続のコツです。

## [03:00 - 04:30] 4. 出力結果をリッチにするTips
せっかく公開するなら、見やすさにもこだわりたいですよね。VitePressなら標準装備、または簡単なプラグインで以下のような表現が可能です。

（各Tipsの画像を見せながら）
**目次 (ToC)**：長いメモでも右側に自動で目次が生成されるので、読者が迷子になりません。あるとないと、随分ユーザビリティが違います。

**アラート・コンテナ**：注意や情報など、視覚的に強調したい部分が簡単に作れます。Markdownの特殊な記法を書くだけなので、手間はゼロです。

**コードブロック**：シンタックスハイライトはもちろん、特定の行を強調できたり、ファイル名を表示できたり。技術的な説明にぴったりです。

**MathJax統合**：数式も美しくレンダリングできるので、実装前の理論を書き留めておくのに便利。理系のメモにも対応できるんです。

これらすべてが、Markdownの記法だけで実現できる。これがVitePressの大きな強みです。
